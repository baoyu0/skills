# SOUL.md Injection：让方法论始终生效

SOUL.md（`$HERMES_HOME/SOUL.md`）是 Hermes Agent 的**身份/个性文件**，在每次对话的 system prompt 中 slot #1 位置加载，优先级高于任何 skills。

利用 SOUL.md 注入方法论指令，是实现「始终生效」的零依赖方式——不需要依赖 config.yaml 的 skill 加载机制，不需要 CLI 参数，不需要显式 `/skill`。

## 原理

- SOUL.md 内容直接拼入 system prompt
- 它是自由格式 Markdown，可以包含方法流程、原则、约束
- 对 agent 来说，SOUL.md 的指令优先级等同于用户预设的个性规则
- 不需要修改 Hermes 核心或等待功能更新

## 适用场景

| 场景 | 适合 |
|------|------|
| 方法论规则（拼好码 7 步） | ✅ 完美 |
| 输出格式偏好 | ✅ 完美 |
| 沟通风格指令 | ✅ 完美 |
| 工作流程约束 | ✅ 完美 |
| 工具链指令 | ⚠️ 可以但不如 skill_view 结构好 |
| 动态数据/凭证 | ❌ 不适合（SOUL.md 是静态文件） |

## 最佳实践

1. **保持简洁** — 不超过 20 行。SOUL.md 太长会挤占用户消息空间
2. **用具体指令而非建议** — "必须先按要求做" 而非 "建议参考"
3. **给出口诀/速查表** — 7 步口诀比自然段易记
4. **不和 skill 内容重复** — SOUL.md 注入概要，skill 完整版
5. **放在文件末尾** — 不干扰已有的个性/风格段落
6. **定期审查** — 每个季度确认指令仍准确

## 示例

```markdown
## 方法论 === 拼好码（Code Assembly）
每次新任务，**必须先**按以下流程执行，不走完不得动手：
1. **写清需求** — 目标、输入、输出、约束、验收标准
2. **反向搜索** — 拆能力 → ①官方 → ②社区 → ③平台
3. **评估候选** — stars、许可证、更新日、生产案例
4. **选组合 + 决策记录** — 为什么选、为什么不选、为什么不自研
5. **设计边界** — 适配层、回滚路径
6. **只写胶水代码** — 不重写成熟能力的内部逻辑
7. **验证交付** — 逐条过验收标准
反模式：直接写脚本 × 凭印象选方案 × 先写再换 × 自研一切
关键词：能复用时不重造，能编排时不发明。
```

## 风险

- 方法论条目在每次 system prompt 中都占 token（约 200 tokens）
- 修改后需要新对话生效（不动态重载）
- 不适合存放长文档或参考数据（那是 skills 的职责）
