# 拼好码：Code Assembly

**能复用时不重造，能编排时不发明。**

拼好码要求从「实现者心态」转向「整合者心态」：不是看到需求就让 AI 直接自研，而是先识别已有成熟能力、评估成熟度、设计适配边界，再用最少自研完成业务闭环。

成熟生态承担通用复杂度，胶水代码连接业务流程，自研只服务不可替代的业务差异。

---

## 思维转变

| 从（实现者） | 到（整合者） |
|-------------|-------------|
| "这个功能我写一下" | "这个功能有没有现成的 CLI/包/SDK？" |
| "先跑起来再说" | "先搜方案，评估完再动手" |
| "自己写更可控" | "用成熟工具，出问题时上游修" |
| "就一个小功能不用查" | "再小的功能也可能有成熟库" |
| "这个工具我熟悉" | "确认一下有没有更新的替代品" |

## Quick Start

刚加载本 skill？直接照着做：

```
1. 读需求 → 2. 拆能力搜方案 → 3. 评估 → 4. 选+记录 → 5. 定边界 → 6. 写胶水 → 7. 验证
```

**前 30 秒**：判断任务是否小于 30 秒（见 When to Skip）。是→跳过。否→继续。
**前 2 分钟**：拆能力、跑搜索、扫一眼 stars 和更新日。
**前 5 分钟**：评估矩阵、选方案、定边界。
**之后**：只写胶水代码，不做自研。

> 第一次用不熟练很正常。强迫自己走完 3 次完整流程，直觉就养成了。

## When to Skip（何时可以跳过本流程）

不是所有任务都要走完整 7 步。以下场景允许跳过：

| 场景 | 跳过级别 | 理由 |
|------|----------|------|
| 重命名文件、改参数、修 typo | 全部跳过 | 不需要工具选型 |
| 已知唯一方案的常规操作（如 `git commit`） | 跳过搜索+评估 | 方案已确定 |
| 用户明确指定工具/方案 | 跳过搜索+评估+选型 | 用户已决策 |
| 紧急修复（生产宕机） | 仅走验证 | 先止血再优化 |

**任何带「搜方案」价值的任务都不允许跳过。** 不确定时走完整流程。

## 7 步工作流

### 1. 写清需求

先和用户对齐以下内容，未明确的先问清楚：

| 维度 | 内容 | 示例 |
|------|------|------|
| 目标 | 要解决什么问题 | "把 Obsidian markdown 发布到 Halo 博客" |
| 输入 | 提供什么数据/文件/URL | .md 文件路径、frontmatter 元数据 |
| 输出 | 期望得到什么结果 | 发布成功的文章 URL |
| 约束 | 平台、环境、性能、预算、时间 | Windows 10, 无 docker, 预算 0 |
| 验收标准 | 什么算"做完" | 文章发布成功，页面排版正确，图片正常显示 |

**产出**：一份结构化的需求陈述，贴在任务计划开头。

### 2. 反向搜索

根据需求拆出能力领域，按优先级三层搜索。

**搜索次序（硬规定，不可跳过）：**

```
① 官方能力 → ② 社区成熟工具 → ③ 平台服务
```

**① 官方能力（第一优先级）**
- `skills_list()` + `skill_view(name)` — 已有的 skills 是否能直接覆盖
- MCP 服务器暴露的工具
- 官方 CLI / SDK / API（如 `@halo-dev/cli`、`gh`、`yt-dlp`、`ffmpeg` 等）
- 操作系统/平台自带能力（PowerShell、注册表、WMI、COM 接口）

**② 社区成熟工具（第二优先级）**
- `web_search_plus(query="<能力> CLI OR npm OR Python library comparison 2025 2026")`
- `mcp_github_search_repositories(query="<能力> language:<lang> stars:>500")`
- `terminal(command="pip search <能力>" 或 "npm search <能力>")`
- 事实标准：社区共识的"最佳实践"工具

**③ 平台服务（最后优先级）**
- 云服务 API、托管平台、SaaS
- 一般需要注册/付费，仅在①② 都找不到时考虑

**搜索模板（可直接复制的命令）：**

```
# 搜 CLI 工具
web_search_plus(query="<能力> CLI tool OR command line 2025")

# 搜 npm 包
web_search_plus(query="<能力> npm package OR node library 2025 2026")

# 搜 Python 包
web_search_plus(query="<能力> Python library OR pip package 2025 2026")

# 搜 GitHub 仓库（按 stars）
mcp_github_search_repositories(query="<能力> language:python stars:>500 sort:stars")

# 搜横向对比
web_search_plus(query="<能力> vs OR comparison OR alternatives 2025")
```

### 3. 评估候选

对每个候选方案，逐项评估。**不满足判定标准的果断放弃，不要勉强。**

| 维度 | 检查项 | 判定标准 |
|------|--------|----------|
| 维护状态 | 最后提交日期、Issue 响应、发版频率 | 最近 6 个月有实质性更新 |
| 许可证 | LICENSE 文件 | MIT/Apache-2.0 优先，GPL 看场景 |
| 文档质量 | README、API 文档、示例代码、教程 | 有完整 README + 可运行的快速开始 |
| 生产案例 | 已有用户、下游依赖（反向依赖）、GitHub 引用 | stars > 500 或知名项目在使用 |
| 生态兼容 | 与已有工具链整合难度、依赖冲突 | 不引入冲突依赖、不要求特殊环境 |
| 替换风险 | 迁移成本、vendor lock-in、升级兼容性 | 适配器模式可隔离，1 天内可切走 |
| 接入成本 | 安装步骤数、认证配置复杂度、踩坑可能 | 5 步以内可用，不踩坑 |

**评估产出**：对每个候选写一行判断，格式：

```
- <工具名> — ✅ 采用（原因）/ ❌ 放弃（原因）
```

### 4. 选择组合

确定最终采用的工具/仓库组合，**并记录决策理由**。

决策记录模板（拷贝到任务计划中）：

```
## 方案决策

### 采用

- **<工具/仓库 A>** — 做什么
  - 为什么选它：<相比竞品的核心优势>
- **<工具/仓库 B>** — 做什么
  - 为什么选它：<...>

### 不选

- **<竞品 X>** — 被否原因
  - 维护停滞 / 功能缺失 / 许可证问题 / 生态不兼容
- **<竞品 Y>** — 被否原因
  - <...>

### 不自研原因

- 核心复杂度由成熟生态承担，差异化价值不足以覆盖自研和维护成本
- 即使只算"写代码的时间"，写出来 + 测稳定 + 修 bug 也远超评估和集成的时间
```

**「为什么不选 X」比「为什么选 Y」更有长期价值。** 6 个月后的自己会感谢这个记录。

### 5. 设计边界

固定接口契约，防止成熟能力升级时牵连业务代码：

- **输入输出契约** — 数据模型、类型定义、JSON Schema / Pydantic model
- **接口签名** — 调用方式、参数、返回值（用适配器/包装器模式隔离第三方 API）
- **错误处理** — 重试策略（退避）、降级方案（用次优方案替代）、超时设置
- **依赖隔离** — 包装层（Wrapper/Adapter），第三方失效时业务代码不受影响
- **回滚路径** — 如何在 5 分钟内切回旧方案（保留旧代码路径、功能开关、版本标记）

**边界设计不是过度工程，是投资。** 升级或替换时省 10x 时间。

### 6. 生成胶水代码

**只写以下内容：**

| 类别 | 内容 | 示例 |
|------|------|------|
| 连接代码 | 初始化和调用成熟工具 | `subprocess.run(["ffmpeg", ...])` / `client = boto3.client(...)` |
| 适配封装 | 隔离第三方依赖的接口层 | `class VideoConverter: ...` 内部封装 ffmpeg |
| 编排逻辑 | 多工具串并联流程 | 下载 → 转码 → 上传 → 发布 |
| 配置管理 | 加载环境变量、配置文件 | `.env` + `pydantic-settings` |
| 业务规则 | 你业务独有的逻辑 | 文章标题规范、标签生成规则 |
| 测试用例 | 验证胶水代码的正确性 | unit test + integration test |

**绝不写：** 成熟能力本身的内部逻辑。
- 不要在项目中拷贝 CLI 的实现代码
- 不要重写 SDK/API 的内部逻辑
- 不要 fork 后魔改（除非需要提 PR 回上游）

### 7. 验证交付

逐项检查清单（**每一项都必须过，不满足不准说"完成"**）：

- [ ] 类型检查通过（TypeScript / mypy / Pydantic schema validation）
- [ ] 集成测试覆盖关键路径
- [ ] CI 门禁通过（如果有）
- [ ] **验收标准逐条满足**
- [ ] 留下证据（输出文件、截图、响应日志、文章 URL）
- [ ] 替换方案已记录在决策文档中
- [ ] 回滚路径已验证可行

## 实战案例

**需求**：把本地 Markdown 处理（编号、清理）后发布到 Halo 博客。

**搜索过程**：
1. ① 官方能力 → 找到 `@halo-dev/cli` 的 `import-markdown` 命令
2. 评估 → ✅ 官方维护、支持批量、支持 frontmatter
3. **选择**：官方 CLI 代替自研 Python requests 发布脚本
4. **胶水代码**：只写本地 markdown 处理（cleanup + auto-number），发布走官方 CLI
5. **结果**：删除 200+ 行自研代码，稳定性大幅提升

**关键决策**：最初用 Python 脚本 curl Halo API，后来发现官方 CLI 存在后立即迁移。

### 案例 2：X Article 视频下载

**需求**：从 X/Twitter 文章页面下载内嵌视频。

**搜索过程**：
1. ① 官方能力 → yt-dlp 不支持 X Article（只支持普通推文）
2. ② 社区工具 → gallery-dl 只支持图片，videodl 有 bug
3. ③ 搜索 GitHub Issues → 确认目前无任何 CLI 工具支持
4. **决策**：放弃自动化 → 使用用户已有的 猫抓 扩展 + 手动下载
5. **边界**：用 obu CDP 提取 poster 封面作为视觉兜底

**关键教训**：有些场景确实没有成熟方案，及时识别并选择务实的手动方案，比强行自研更有价值。

### 案例 3：BrowserAct 评估后放弃

**需求**：增强浏览器自动化能力。

**搜索过程**：
1. 找到 BrowserAct（browser-act/skills）
2. **评估**：
   - 核心功能 obu 已覆盖 90%
   - stealth-extract 需付费 API key
   - 用户已有完整工具栈（obu + OpenCLI + yt-dlp + 猫抓）
3. **决策**：❌ 放弃，不安装
4. **记录原因**：锦上添花，不解决新问题，增加复杂度

**关键教训**：30 分钟的评估省了 2 天的试错。评估并拒绝，也是拼好码的一部分。

### 案例 4：Karing GitHub 下载修复

**需求**：Karing 代理导致 GitHub Release 下载失败。

**搜索过程**：
1. 分析错误 → 302 重定向到 CDN 域名（release-assets.githubusercontent.com）
2. 理解根因 → `geosite:github` 只覆盖 github.com，不覆盖 CDN 域名
3. **解决方案**：在 Karing 路由规则中加两条 `domain_suffix`
4. **胶水代码**：零代码，只改配置

**关键教训**：有些"问题"不需要代码解决，理解系统架构后改配置就行。过早写代码反而掩盖根因。

## 执行协议

> 以下协议是这个 skill 的灵魂，不遵守就等于没用这个 skill。

1. **搜索不可跳过** — 不论任务大小，必须先走完 ①→②→③ 搜索链。禁止凭印象直接选方案
2. **搜索失败先拆解** — 如果 X 搜不到，把 X 拆成 Y+Z 分别搜。X = 文本提取 + 翻译 + 格式化，分别搜
3. **评估不可省略** — 每个候选必须逐项过评估矩阵。不看 stars、更新日期、许可证就直接用 = 赌博
4. **记录不可偷懒** — 方案决策必须写下来，包括「为什么不选 X」。写的过程本身能发现盲区
5. **边界不可跳过** — 必须设计适配层 + 回滚路径。没有回滚路径的方案不是可交付方案
6. **每步可回退** — 任何一步评估不通过，回到步骤 2 继续搜。不硬推进
7. **验证才等于完成** — 验收标准逐条过完才叫"做完"。没验证的交付 = 猜

## 反模式索引

| ❌ 错误 | ✅ 正确 | 根因 |
|---------|---------|------|
| 看到需求直接写 Python 脚本 | 先查官方 CLI 和现有 Skills | 低估搜索 ROI，高估自研速度 |
| 用 browser_navigate 去已登录站点 | 用 obu 真实 Chrome（已登录） | 忽略用户已有登录态可复用 |
| 用 Playwright 做自动化 | 用 obu CDP / OpenCLI adapter | 忘记用户已有 Chrome 登录态 |
| 用 sed/awk 改文件 | 用 patch 工具 | 不用原生文件编辑工具 |
| 直接 curl API 而不查官方 SDK | 先找 npm/pip/uv 包 | 查包管理器比读 API 文档快 |
| 凭印象推荐方案 | 先搜当前最新状态 | 6 个月前信息可能已过时 |
| "这个我熟，不用查" | 再确认一次版本和更新日期 | 工具迭代快，假设过期 |
| "就一个函数而已" | 搜一下包名 + 这个函数名 | 再小的功能也可能有成熟库 |
| 先自研、发现不行再搜 | 先搜，评估完再动手 | 自研到一半再切换成本高 10x |
| fork 魔改不走 PR | 需要改动时先提 issue 或 PR | fork 分支无人维护，失血而死 |

## 已知陷阱

1. **「先写吧，不行再换」** — 逆方向。写完再换的代价是搜索评估的 10x。必须先搜索再动笔。
2. **拆解不彻底** — 用户要的是 X，X 可能没有现成的，但 X = Y+Z，Y 和 Z 可能都有。深入拆分能力领域，不要卡在 X 搜不到就放弃。
3. **"用 CLI 不如自己写更可控"** — CLI 出错时由上游维护者负责，自研代码出错时你负责。CLI 天然有回滚路径（版本升降），自研没有。
4. **只搜高质量方案** — 优先看 stars>500 / 最近 6 个月有更新 / 有知名用户的项目。不花时间评估无人维护或 Issue 长期未回复的仓库。
5. **适配层不是过度设计** — 包装第三方工具的一层接口是投资不是浪费。升级版本、切换竞品时省 10x 时间。没有适配层的代码 = 与第三方实现耦合。
6. **验证不能省略** — 工具文档说支持 = 文档上说支持，实际跑通才是真正支持。每个环节都要实测，特别是边界情况（空输入、网络超时、权限不足）。
7. **选型记录给未来留线索** — "为什么不选 X" 比 "为什么选 Y" 更有价值。6 个月后重看决策记录能避免重新论证。不写 == 重新交学费。
8. **"先搜就对了" 但搜完不记录** — 搜索过程是最容易丢失的信息资产。搜过的关键词、看过的方案、排除的理由，简单记几行就能让下次省一半时间。
9. **看到新工具就想换** — 平台工具流（CLI/SDK 新版本频出）不等于应该立即迁移。评估时要问：现有方案真的不可用吗？迁移收益覆盖迁移成本吗？
10. **忽略回滚路径** — 没有 roolback plan 的方案 = 有隐患的方案。最少的方式：保留旧代码路径 / feature flag / 版本标记。

## 速查表

```
需求 → 拆能力 → 搜①官方 → 搜②社区 → 搜③平台 → 评估 → 选组合 → 定边界 → 写胶水 → 验证
```

每步没通过就回到上一步重新搜，不硬推进。

## Installation & Deployment

This skill supports three deployment modes:

### A. SOUL.md Injection（推荐 — 始终生效）

适用于 Hermes Agent。把 7 步口诀注入 `$HERMES_HOME/SOUL.md`，每次新对话自动生效，无需显式加载。

做法：在 SOUL.md 末尾添加以下内容：

```markdown
## 方法论 === 拼好码（Code Assembly）
每次新任务，**必须先**按以下流程执行，不走完不得动手：
1. **写清需求** — 目标、输入、输出、约束、验收标准
2. **反向搜索** — 拆能力 → ①官方 → ②社区 → ③平台
3. **评估候选** — stars、许可证、更新日、生产案例
4. **选组合 + 决策记录** — 为什么选、为什么不选、为什么不自研
5. **设计边界** — 适配层、回滚路径
6. **只写胶水代码** — 不重写成熟能力内部逻辑
7. **验证交付** — 逐条过验收标准
关键词：**能复用时不重造，能编排时不发明。**
```

### B. Hermes CLI Preload（每次会话指定）

```bash
hermes --skills code-assembly
```

或在会话中使用 `/skill code-assembly` 加载。

### C. External Skills CLI（跨 Agent）

```bash
npx skills add baoyu0/code-assembly-skill@code-assembly -g -y
```

适用于 Claude Code 和 Codex。安装后参考 `CLAUDE.md` / `AGENTS.md` 了解加载方式。
