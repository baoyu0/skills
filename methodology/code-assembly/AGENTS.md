# Code Assembly (拼好码)

> 能复用时不重造，能编排时不发明。

为 Codex 等 AI 编程 Agent 设计的方法论。每次编码任务开始时加载本文件。

## 核心流程（不可跳过）

每步不通过就回到上一步重新搜索，不硬推进。

```
需求 → 拆能力 → 搜①官方 → 搜②社区 → 搜③平台 → 评估 → 选组合 → 定边界 → 写胶水 → 验证
```

### 1. 写清需求
明确：目标、输入、输出、约束、验收标准。

### 2. 反向搜索
搜索次序（硬规定）：**① 官方能力 → ② 社区成熟工具 → ③ 平台服务**

- ① 查 CLI/SDK/API/内置工具
- ② 搜 GitHub (stars>500)、npm/pip/cargo、横向对比文章
- ③ 云服务 API、SaaS（需要付费，仅①②找不到时考虑）

### 3. 评估候选
检查：更新日期、许可证、stars、文档、生产案例、依赖冲突。
不满足标准的果断放弃。

### 4. 选择组合
记录决策：为什么选 A、为什么不选 B、为什么不自研。
**「为什么不选 X」比「为什么选 Y」更有长期价值。**

### 5. 设计边界
固定接口契约：输入输出模型、适配器包装、错误处理、回滚路径。

### 6. 生成胶水代码
只写连接代码、适配封装、编排逻辑、配置、业务规则、测试。
**绝不重新实现成熟能力本身的内部逻辑。**

### 7. 验证交付
- [ ] 类型检查通过
- [ ] 集成测试覆盖关键路径
- [ ] 验收标准逐条满足
- [ ] 留下证据（输出文件、截图、响应日志）
- [ ] 回滚路径已验证可行

## When to Skip

| 场景 | 跳过级别 |
|------|----------|
| 重命名/typo/改参数 | 全部跳过 |
| 已知唯一方案的常规操作 | 跳过搜索+评估 |
| 用户已指定方案 | 跳过搜索+评估+选型 |
| 生产紧急修复 | 仅走验证 |

**不确定时走完整流程。**

## Quick Reference

成功标准：任务结束后，你能回答这三个问题——
1. 我搜过哪些方案？（关键词+渠道）
2. 为什么选这个方案？（决策理由）
3. 如果方案不可用，回滚路径是什么？

答不出来 = 没走完整流程。
